# Полное руководство по использованию ZAGARAMI

Актуально для текущего репозитория в директории `/Users/nikitamysnik/Desktop/progs/stones`.

Публичный бренд системы — ZAGARAMI. В репозитории, env-переменных и части служебных идентификаторов сохраняется внутреннее имя `stones`.

## 1. Назначение системы

ZAGARAMI — это система для:

- публичной витрины уникальных камней;
- работы HQ с каталогом, партиями, складом и пользователями;
- работы франчайзи с задачами на сбор;
- публикации цифрового паспорта конкретного камня;
- базового учета финансов партнера.

Система покрывает полный путь:

1. HQ создает каталог и задачу на сбор.
2. Франчайзи принимает задачу и создает партию.
3. HQ получает партию, принимает item, загружает фото/видео и завершает партию.
4. HQ выводит item в онлайн-продажу.
5. Покупатель видит товар, отправляет заявку и позже может открыть цифровой паспорт.

## 3. Быстрый локальный запуск

### 3.1 Установка

```bash
npm install
```

### 3.2 Настройка `.env`

Минимальный пример:

```env
PORT=3001
CLIENT_URL=http://localhost:5173
DATABASE_URL="mysql://stones:stones@localhost:3307/stones?connection_limit=20&pool_timeout=30"
ACCESS_TOKEN_SECRET="change_me_access"
REFRESH_TOKEN_SECRET="change_me_refresh"
VITE_HOST=127.0.0.1
VITE_PORT=5173
VITE_API_TARGET=http://127.0.0.1:3001
```

### 3.3 Подготовка базы

```bash
npm run db:migrate
npm run db:seed:languages
npm run db:seed
```

### 3.4 Запуск

```bash
npm run dev
```

После запуска:

- frontend: `http://localhost:5173`
- backend: `http://localhost:3001`

## 9. Работа HQ

## 9.1 Dashboard

- партии в `TRANSIT`;

## 9.2 Orders

Доступно:

- `ADMIN`

Что можно:

- редактировать клиентские поля у незакрытых заказов;
- вести внутреннюю заметку;
- менять статус по допустимым переходам.
- мягко скрывать заказ из интерфейса без физического удаления из БД.

## 9.3 Locations

Можно:

- создавать локации;
- редактировать координаты, изображение и базовый перевод;
- редактировать переводы;
- мягко скрывать локации из интерфейса. Связанные товарные шаблоны и партии тоже уходят из UI, но остаются в БД.

## 9.4 Products

Можно:

- создавать товарные шаблоны;
- редактировать цену, фото, коды и переводы;
- публиковать/снимать с публикации;
- задавать ссылки Wildberries и Ozon;
- создавать задачу на сбор по товару;
- мягко скрывать товарный шаблон из интерфейса без физического удаления.

### 9.4.1 Soft delete и очистка каталога

- Для разовой очистки текущей БД от каталога и связанных операционных сущностей используйте `npm run db:soft-delete:business`.
- Скрипт не удаляет строки физически: проставляет `deleted_at` у локаций, товаров, партий, item, заказов на сбор и заказов покупателей.
- Публичные клоны для скрытых сущностей перестают открываться и возвращают `404`, а не `500`.

## 9.5 Acceptance

Рабочий путь:

1. Открыть партию в `TRANSIT` или `RECEIVED`.
2. Если партия еще в пути, перевести ее в `RECEIVED`.
3. Проверить item внутри партии.
4. Принять item или отклонить.
5. Подготовить итоговые фото и видео.
6. При необходимости распечатать QR.
7. Завершить партию только после полной media-готовности.

Дополнительно:

- в складской структуре администратор может мягко скрыть партию;
- скрытая партия и связанные item исчезают из интерфейса, но остаются в БД.
- в карточке партии видны счетчики `Фото готовы`, `Видео готовы`, `Media полностью`;
- из карточки партии доступны быстрые переходы в `Photo Tool`, `Video Tool` и `QR Print`.

## 9.6 Photo Tool

Используется HQ для подготовки финальных item-фото перед финализацией партии.

Сценарий:

1. открыть `/admin/photo-tool/:batchId`
2. загрузить один или несколько файлов изображений
3. при необходимости сменить сортировку, инвертировать список или назначение
4. проверить, что каждая позиция `001..NNN` получила уникальную фотографию
5. сохранить назначения в backend

Особенности:

- Photo Tool работает только для партии в статусе `RECEIVED`;
- у всех item в партии должен быть заполнен `item_seq`;
- можно использовать уже сохраненные фото партии или загрузить новые image-файлы;
- номер под фото соответствует `item_seq`, а не `serial_number`;
- лишние фото допустимы, но без полного покрытия партии сохранить нельзя;
- несохраненный черновик восстанавливается после перезагрузки страницы;
- если другой оператор изменил фото партии, save будет отклонен до обновления страницы.

## 9.7 Video Tool

Используется HQ для подготовки финальных item-видео.

Сценарий:

1. открыть `/admin/video-tool/:batchId`
2. выбрать исходное видео партии
3. дождаться проверки helper и совместимости исходника
4. разрезать таймлайн на товарные фрагменты `001..NNN`
5. создать или продолжить export-session
6. дождаться рендера финальных `.mp4`
7. загрузить готовые ролики обратно в backend

Особенности:

- Video Tool проверяет локальный helper и версию протокола `stones-video-export-helper-v2`;
- черновик нарезки и текущая session восстанавливаются после перезагрузки страницы;
- export-session имеет статусы `OPEN`, `UPLOADING`, `COMPLETED`, `FAILED`, `CANCELLED`, `ABANDONED`;
- при зависшей или частично загруженной session доступен `retry-tail` только для отсутствующих роликов;
- после `CANCELLED` следующая попытка создает новую session.

## 9.8 QR Print

HQ-печать QR работает через:

- `/api/batches/:batchId/qr-pack`
- `/admin/qr/print?batchId=<ID>&mode=all`
- `/admin/qr/print?batchId=<ID>&mode=selected&ids=<itemId,itemId>`

В QR-пакет попадают только item, которые уже могут иметь публичный паспорт.

Текущий UI-сценарий:

1. открыть сервис из `Acceptance`, `Products` или из сайдбара `QR-печать`
2. выбрать товар-шаблон и партию
3. переключить режим `Печатать все` или `Выбрать вручную`
4. настроить размеры этикетки, QR, шрифты и состав полей
5. при включенном поле `Свое поле` заполнить текст прямо в этикетках документа
6. отправить документ в `Печать / PDF`

Поддерживаемые поля этикетки:

- `Название`
- `Локация`
- `Координаты`
- `Время`
- `Серийный номер`
- `Свое поле`

## 11. Публичный цифровой паспорт

### URL и API

- страница: `/clone/:serialNumber`
- данные: `GET /api/public/items/:serialNumber`
- QR PNG: `GET /api/public/items/:serialNumber/qr`

### Когда паспорт доступен

- batch в `RECEIVED` или `FINISHED`
- item не `REJECTED`
- `serial_number` не legacy

### Что видно в паспорте

- название товара;
- описание;
- дата и время сбора;
- координаты;
- фото;
- видео;
- серийный номер.

Источник media в API:

- фото возвращается как `item_photo_url`, а если его нет — как fallback из `photo_url`;
- видео возвращается из `item_video_url`.

## 12. Публичная активация

Endpoint:

- `POST /api/public/items/:serialNumber/activate`

Он:

- ставит `ACTIVATED`;
- пишет `activation_date`;
- отмечает item как `is_sold`.

Он не:

- создает бухгалтерскую проводку;
- меняет баланс франчайзи.

## 14. Частые проблемы

### 14.1 Не поднимается backend

Проверьте:

- заполнены ли `ACCESS_TOKEN_SECRET` и `REFRESH_TOKEN_SECRET`
- заполнены ли при интеграции доставки `CDEK_CLIENT_ID` и `CDEK_CLIENT_SECRET`
- доступна ли MySQL
- корректен ли `DATABASE_URL`

### 14.2 Не работает buyer login/register

Проверьте:

- используется ли `username` для buyer-регистрации
- не занят ли уже логин
- корректно ли сохранены access/refresh tokens

### 14.3 Photo Tool не сохраняет назначения

Проверьте:

- batch уже в `RECEIVED`
- у всех item есть `item_seq`
- каждая позиция партии покрыта уникальной фотографией
- страницу не успел пересохранить другой оператор

### 14.4 Video Tool не завершает экспорт

Проверьте:

- локальный helper отвечает и версия протокола совпадает;
- не выбрана уже `COMPLETED` или `CANCELLED` session;
- для частичной дозагрузки используется `retry-tail`;
- исходник успешно импортирован в helper.

### 14.5 Партия не финализируется

Проверьте:

- batch уже в `RECEIVED`
- нет активной video-задачи
- у каждого item есть фото и видео

### 14.6 QR не открывается

Проверьте:

- item не `REJECTED`
- batch в `RECEIVED` или `FINISHED`
- у item есть актуальный `serial_number`
