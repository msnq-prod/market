# Правила развития репозитория Stones

Документ фиксирует реальные правила, которые имеет смысл соблюдать в текущем проекте.

## 1. Источники истины

- схема данных: `prisma/schema.prisma`
- миграции: `prisma/migrations/*`
- seed-данные: `prisma/seed.ts`, `prisma/seed_languages.ts`
- API-контракты: текущие route-модули в `server/routes/*`
- роли и UI-маршруты: `src/App.tsx`, `src/admin/*`, `src/partner/*`

Если документация расходится с кодом, приоритет у кода.

## 2. База данных и миграции

- любые изменения схемы только через `prisma migrate`
- ручные DDL-правки в production запрещены
- enum-статусы нельзя переименовывать без миграции и проверки всех зависимостей
- после изменения схемы нужно проверить seed и документацию

## 3. API и контракты

- не менять публичные контракты без явной необходимости
- ошибки API возвращать в текущем стиле: `{ error: string }`
- если меняется запрос или ответ endpoint, нужно обновить:
  - frontend, который его использует
  - релевантные документы в `docs/`
  - тесты, если они завязаны на контракт

## 4. Бизнес-статусы

Особо чувствительные группы статусов:

- `OrderStatus`
- `CollectionWorkflowStatus`
- `BatchStatus`
- `ItemStatus`
- `SalesChannel`

Для этих статусов:

- не менять значения "по месту"
- не переиспользовать старое значение для нового смысла
- перед изменением проверять backend, frontend, seed, e2e и docs

## 5. Аутентификация и роли

- staff и franchisee-аккаунты создаются через защищенный `POST /api/users`
- buyer-аккаунты создаются через открытый `POST /auth/register`
- доступ в HQ и partner UI должен оставаться разделенным по ролям
- выдачу роли `ADMIN` держать минимальной

## 6. Frontend

- основной язык интерфейса по умолчанию — русский
- не ломать маршруты:
  - `/`
  - `/clone/:serialNumber`
  - `/admin/login`
  - `/admin/*`
  - `/partner/login`
  - `/partner/*`
- при добавлении новых сценариев учитывать mobile width, loading и error state

## 7. Медиа и видео

- фото и видео item должны оставаться привязанными к конкретному `Item`, а не к `Product`
- публичный цифровой паспорт строится только вокруг `serial_number`
- batch нельзя финализировать без полного набора `item_photo_url` + `item_video_url`
- video workflow нужно документировать отдельно, если меняется операторский сценарий

## 8. Документация

После существенных изменений обновлять:

- `README.md`
- `docs/BUSINESS_LOGIC_RU.md`
- `docs/SYSTEM_USAGE_GUIDE_RU.md`
- ролевые гайды, если затронуты соответствующие контуры
- `docs/DOCKER_RU.md` и `docs/OPERATIONS.md`, если меняется deploy/ops
- `docs/TEST_CREDENTIALS_AND_TECH_INFO_RU.md`, если меняются seed-аккаунты или test data

## 9. Проверки перед сдачей

Минимум по возможности:

```bash
npm run lint
npm run build
```

Если затронуты e2e-сценарии или пользовательские потоки:

```bash
npm run test:e2e
```

Если проверка не запускалась, это нужно явно указывать в отчете.

## 10. Безопасность

- секреты только через env
- пароли только в хэшированном виде
- нельзя добавлять в репозиторий реальные приватные ключи, токены и production secrets
- любые локальные runbook-файлы с реальными секретами должны быть заменены на шаблоны перед публикацией
