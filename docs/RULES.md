# Правила разработки

Этот документ фиксирует правила и соглашения, по которым развивается система.

## Источник истины
- Схема базы данных определяется в `prisma/schema.prisma`, все изменения идут только через миграции.
- Ручные DDL-правки в проде запрещены.
- Каждое изменение схемы должно иметь понятное имя миграции и краткое обоснование в PR/коммите.

## Окружения
- Отдельные окружения: `dev`, `staging`, `prod`.
- Миграции обязаны отрабатываться на `staging` перед `prod`.
- Данные продакшена не используются в `dev`.

## Миграции
- Использовать только `prisma migrate`.
- Новые таблицы должны иметь `created_at` и `updated_at` (если нет четкого исключения).
- Любые ломающие изменения требуют плана совместимости и отката.

## Именование
- Единый стиль имен для таблиц и полей: `snake_case`.
- Первичный ключ всегда `id` (UUID по умолчанию), кроме доменов с явной потребностью в числовом ID.
- Внешние ключи именуются `<entity>_id` (пример: `location_id`).

## Индексы
- Каждый внешний ключ обязан иметь индекс.
- Уникальные поля оформляются `UNIQUE` (пример: `email`, `slug`, `code`).
- Композитные индексы добавляются под реальные фильтры (пример: `language_id + entity_id`).

## Целостность данных
- Где возможно, использовать `NOT NULL`.
- Поддерживать ссылочную целостность с корректными `ON DELETE`/`ON UPDATE`.
- Не хранить вычисляемые значения без явной причины (производительность/аудит).

## Локализация
- Пользовательские тексты живут в таблицах переводов.
- Переводы обязательно связаны с `language_id` через FK.

## Доступ к данным
- Запрещено обращаться к Prisma напрямую из роутов.
- Весь доступ к данным идет через сервисный слой (например `ProductService`, `LocationService`).

## Производительность
- При росте чтений использовать read-replicas.
- Справочные данные (языки, категории) кэшируются в Redis.
- В листингах избегать больших `include`; использовать `select` и фильтры.

## Наблюдаемость
- Логировать длительность и статус каждого запроса.
- Включить slow query log в MySQL.
- Отслеживать p95 латентность по ключевым эндпоинтам.

## Тестирование
- Миграции проходят в CI на каждом изменении схемы.
- Держать минимальный нагрузочный baseline и обновлять его при смене паттернов.

## Безопасность
- Все секреты только через окружение.
- Пароли хранятся только в виде хэшей.
- Ввод валидируется на уровне API.
