# Бизнес-логика Stones

Документ описывает текущую бизнес-логику системы по фактической реализации в коде.

Для заказчика простым языком система описана в [BUSINESS_PROCESSES_AUDIT_RU.md](./BUSINESS_PROCESSES_AUDIT_RU.md). Здесь зафиксирована более предметная модель: роли, статусы и реальные переходы.

## 1. Контуры системы

Система состоит из трех основных контуров:

- `Public Front` — витрина, buyer-аккаунт, корзина, заявка на покупку, цифровой паспорт.
- `Admin HQ` — каталог, заказы, приемка, склад, публикация, QR, контент, пользователи.
- `Partner Core` — задачи на сбор, партии, партнерский дашборд, финансы.

## 2. Роли

- `ADMIN` — полный доступ к HQ-контуру.
- `MANAGER` — HQ-операции без очереди заказов сайта.
- `SALES_MANAGER` — только заказы покупателей в `/admin/orders`.
- `FRANCHISEE` — партнерский кабинет и свои партии/финансы.
- `USER` — покупатель публичной витрины.

## 3. Ключевые сущности

### Локация

Точка происхождения товара на карте. Используется на витрине и в каталоге.

### Product

Товар-шаблон. Это не конкретный камень, а тип товара, по которому:

- создаются задачи на сбор;
- формируются партии;
- отображается публичная карточка товара.

### CollectionRequest

Внутренняя задача на сбор для франчайзи. Связывает HQ и partner-контур.

### Batch

Физическая партия камней, отправленная франчайзи в HQ.

### Item

Отдельный уникальный экземпляр. Именно к `Item` привязан:

- `serial_number`
- публичный цифровой паспорт
- QR-код
- фото и видео
- статус продажи/активации

### Order

Заявка покупателя с публичного сайта. В текущей версии это не оплата, а заказ-заявка.

### Soft delete

`Location`, `Product`, `Batch`, `Item`, `CollectionRequest` и `Order` могут быть скрыты через `deleted_at`.

Это означает:

- запись не показывается в интерфейсах и рабочих API;
- запись не удаляется физически и может быть восстановлена напрямую в БД;
- публичный цифровой паспорт для скрытого item или для item со скрытым `Batch/Product/Location` становится недоступен по `404`.

## 4. Статусы и смысл

### 4.1 Заказы покупателей

- `NEW` — новая заявка
- `IN_PROGRESS` — менеджер взял в работу
- `COMPLETED` — обработана и закрыта
- `CANCELLED` — отменена

Допустимые переходы:

- `NEW -> IN_PROGRESS`
- `NEW -> CANCELLED`
- `IN_PROGRESS -> COMPLETED`
- `IN_PROGRESS -> CANCELLED`

### 4.2 Заказы на сбор

- `OPEN` — доступен для принятия
- `IN_PROGRESS` — франчайзи взял в работу
- `IN_TRANSIT` — партия создана и отправлена
- `RECEIVED` — HQ получил партию
- `IN_STOCK` — партия полностью доведена до складского результата
- `CANCELLED` — отменено

### 4.3 Партии

- `DRAFT` — технический статус модели данных
- `TRANSIT` — партия в пути
- `RECEIVED` — HQ получил партию
- `ERROR` — проблемная партия / служебное исключение
- `FINISHED` — партия завершена и доведена до складского состояния

Текущий основной рабочий маршрут:

- `TRANSIT -> RECEIVED -> FINISHED`

### 4.4 Item

- `NEW` — создан, еще не прошел HQ-проверку
- `REJECTED` — отклонен на приемке
- `STOCK_HQ` — на складе HQ
- `STOCK_ONLINE` — доступен в онлайн-продаже
- `ON_CONSIGNMENT` — legacy/seed-сценарий консигнации
- `SOLD_ONLINE` — продан онлайн
- `ACTIVATED` — активирован конечным владельцем

## 5. Путь товара внутри системы

## 5.1 Каталог и подготовка

HQ:

1. создает локацию;
2. создает товар-шаблон;
3. при необходимости публикует товар на витрине;
4. создает задачу на сбор.

Если HQ скрывает `Location` или `Product`, связанные сущности тоже выводятся из интерфейсов через soft delete-каскад.

## 5.2 Принятие задачи на сбор

Франчайзи:

1. видит задачи в статусе `OPEN`;
2. принимает задачу;
3. задача переходит в `IN_PROGRESS`.

## 5.3 Создание партии

Франчайзи:

1. открывает задачу в работе;
2. вводит координаты, дату и время сбора;
3. завершает задачу созданием партии.

Система автоматически:

- создает `Batch` со статусом `TRANSIT`;
- создает нужное число `Item`;
- генерирует `temp_id`, `item_seq`, `serial_number`;
- переводит `CollectionRequest` в `IN_TRANSIT`.

На этом этапе финальные видео не загружаются.

## 5.4 Получение партии в HQ

HQ:

1. открывает приемку;
2. переводит партию из `TRANSIT` в `RECEIVED`;
3. связанный `CollectionRequest` получает статус `RECEIVED`.

## 5.5 Приемка каждого item

HQ:

1. ищет item по `temp_id` внутри партии;
2. принимает item или отклоняет.

Результат:

- `accept` -> `STOCK_HQ`
- `reject` -> `REJECTED`

При отклонении создается запись в `AuditLog`.

## 5.6 Фото, видео и QR

После получения партии HQ:

- использует Photo Tool только для партии в `RECEIVED` и только если у всех item есть `item_seq`;
- открывает Photo Tool и назначает фото по `item_seq` (`001`, `002`, `003`);
- может вручную скорректировать номер под каждой фотографией, если автораздача по порядку файлов не подходит;
- не может сохранить фото частично и получает защиту от перезаписи через `base_photo_state_token`;
- подготавливает итоговые video-файлы по item через export-session и локальный helper;
- может продолжить незавершенную video-session, дозагрузить только отсутствующие ролики через retry-tail или отменить текущую session;
- печатает QR через HQ-only поток;
- может открывать staff-инструменты склада и карточки item.

## 5.7 Финализация партии

Партия может быть завершена только если:

- batch находится в `RECEIVED`;
- не идет активная video processing / video export операция;
- у каждого item есть `item_photo_url` и `item_video_url`.

Тогда система:

- переводит batch в `FINISHED`;
- переводит `NEW` item в `STOCK_HQ`;
- переводит `CollectionRequest` в `IN_STOCK`.

## 5.8 Вывод в продажу

В текущем MVP HQ может распределить только item из `STOCK_HQ`.

Текущий рабочий сценарий:

- `STOCK_HQ -> STOCK_ONLINE`

Поддерживаемые онлайн-каналы в API:

- `MARKETPLACE`
- `DIRECT_SITE`

Но текущий HQ UI использует сценарий `MARKETPLACE`.

`OFFLINE_POINT` в модели данных есть, но новый перевод в этот канал сейчас заблокирован в allocation endpoint.

## 6. Публичная витрина и покупка

Покупатель может:

- открыть витрину;
- смотреть локации и товары;
- добавлять товары в корзину;
- зарегистрировать buyer-аккаунт по `username + password`;
- войти в аккаунт;
- отправить заявку на покупку.

Важное ограничение:

- кнопка оплаты не проводит реальный платеж;
- `POST /api/orders` создает заявку в статусе `NEW`.

## 7. Работа менеджера продаж

`ADMIN` и `SALES_MANAGER`:

- видят очередь заказов;
- ищут заявки по id, имени, логину, телефону, email, адресу;
- меняют статус;
- ведут `internal_note`.

Внутренние заметки:

- видны staff;
- не возвращаются покупателю в `GET /api/orders/my`.

## 8. Цифровой паспорт

Публичный паспорт строится вокруг `Item`, а не `Product`.

Публичные URL:

- страница: `/clone/:serialNumber`
- API карточки: `/api/public/items/:serialNumber`
- QR PNG: `/api/public/items/:serialNumber/qr`

Паспорт доступен, только если:

- `serial_number` не является legacy token;
- batch в `RECEIVED` или `FINISHED`;
- item не в `REJECTED`.

Паспорт возвращает только публичные данные:

- `serial_number`
- `clone_url`
- название и описание товара
- дату и время сбора
- координаты
- фото (`item_photo_url`, а если его нет — `photo_url`)
- item-видео

## 9. Публичная активация

Endpoint:

- `POST /api/public/items/:serialNumber/activate`

Допустимые входные статусы item:

- `ON_CONSIGNMENT`
- `STOCK_ONLINE`
- `SOLD_ONLINE`

Что делает endpoint:

- переводит item в `ACTIVATED`
- заполняет `activation_date`
- ставит `is_sold = true`

Что endpoint больше не делает:

- не создает записи в `Ledger`
- не меняет баланс партнера
- не закрывает финансовый расчет автоматически

## 10. Финансы

Финансовый контур сейчас состоит из:

- поля `balance` у пользователя;
- `commission_rate` у франчайзи;
- истории `Ledger`;
- partner UI для просмотра профиля и ledger.

Типы операций:

- `ROYALTY_CHARGE`
- `SALES_PAYOUT`
- `WITHDRAWAL`
- `MANUAL_ADJ`

Текущее ограничение:

- partner-кабинет показывает финансовые записи, но отдельного полного staff-flow по финансовому закрытию продажи в публичном сценарии нет.

## 11. Что важно помнить при описании системы

- цифровой паспорт привязан к `Item`;
- checkout — это заявка, а не платеж;
- в current MVP новый offline allocation не используется;
- публичная активация не делает финансовые проводки;
- batch и collection request живут синхронно, но имеют разные наборы статусов;
- часть legacy-сценариев все еще есть в seed и модели данных, но не во всех UI-потоках.
