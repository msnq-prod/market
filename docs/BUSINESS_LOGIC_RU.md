# Бизнес-логика Stones

Документ описывает актуальную логику v1 для загрузки товара/камня на сайт, складского учета и цифрового двойника.

## 1. Ключевые сущности

- `Product` — товар-шаблон. Это карточка на сайте и в админке: цена, локация, описания, изображение, коды шаблона, публикация.
- `CollectionRequest` — заказ на сбор партии. Создается из строки товара-шаблона.
- `Batch` — партия, физический результат выполнения одного заказа на сбор.
- `Item` — конкретный камень/экземпляр, который появляется после выполнения партии. В v1 именно `Item` является складской и продаваемой единицей.

Для v1:
- один `Item` = один камень;
- составные товары не поддерживаются;
- публичный цифровой двойник привязан к `Item`, а не к `Product`.

## 2. Товар-шаблон

Администратор или менеджер:
- создает карточку товара;
- задает страну, код локации, код товара, описание локации и описание товара;
- управляет флагом публикации `is_published`.

Пока шаблон не опубликован, он не попадает на публичную витрину.

Публичная витрина показывает:
- только опубликованные шаблоны;
- только агрегированную карточку шаблона;
- наличие по фактическому количеству `Item` со статусом `STOCK_ONLINE` и `is_sold = false`.

Если доступных экземпляров нет, карточка отображается как `Нет в наличии`.

## 3. Заказ на сбор

Заказ на сбор создается из конкретного `Product`.

Ограничения:
- количество только целое число от `1` до `999`;
- один заказ соответствует одной будущей партии;
- заказ можно назначить конкретному партнеру или оставить в общем пуле.

Статусы заказа на сбор:
- `OPEN`
- `IN_PROGRESS`
- `IN_TRANSIT`
- `RECEIVED`
- `IN_STOCK`
- `CANCELLED`

Права:
- `ADMIN` / `MANAGER` создают, редактируют, возвращают в пул, отменяют и удаляют открытые заказы без партии;
- `FRANCHISEE` видит общий пул и может принять доступный заказ.

## 4. Выполнение заказа партнером

Партнер работает только с принятым заказом в статусе `IN_PROGRESS`.

При выполнении заказа партнер:
- указывает координаты;
- указывает дату и время сбора;
- загружает видео партии;
- завершает заказ.

Система автоматически:
- создает `Batch`;
- создает нужное количество `Item`;
- проставляет `item_seq`;
- генерирует `public_token`;
- генерирует `serial_number`.

После выполнения:
- `Batch.status = IN_TRANSIT`
- `CollectionRequest.status = IN_TRANSIT`

## 5. Серийный номер

Серийный номер хранится в `Item.serial_number`.

Формат v1:

`[optional daily_batch_seq][COUNTRY:3][LOCATION:3][ITEM_CODE][DDMMYY][ITEM_SEQ:001-999]`

Пример:

`RUSKHV120326001`

Если в один день по одному шаблону создается вторая и последующие партии, добавляется префикс `daily_batch_seq`.

`temp_id` не является серийным номером:
- это внутренний номер пакетика внутри партии;
- он используется для маркировки и складской сверки.

## 6. Приемка и медиа

После прибытия партии в HQ:
- администратор открывает `/admin/acceptance`;
- администратор переводит партию в `RECEIVED`;
- открывает карточку партии;
- загружает фото по файлам, названным серийным номером;
- открывает локальный HQ-инструмент монтажа видео.

Сопоставление media для фото:
- `ruskhv120326001.jpg` -> фото конкретного `Item`;
- `ruskhv120326002.jpg` -> фото конкретного `Item`.

HQ-монтаж видео:
- основной маршрут: `/admin/video-tool/:batchId`;
- инструмент доступен только для `ADMIN` и `MANAGER`;
- HQ загружает один исходный ролик локально на своем ПК;
- таймлайн режет исходник на `000..NNN`, где `000` — общий вступительный фрагмент;
- каждый финальный ролик собирается как `000 + XXX`;
- итоговый файл получает имя ровно по `Item.serial_number`;
- на удаленный сервер отправляются только готовые финальные MP4;
- после загрузки полного комплекта система назначает `Item.item_video_url` по `serial_number`.

Локальный helper:
- для HQ pilot поставляется как отдельный macOS desktop helper;
- production helper устанавливается один раз и дальше работает как background/menu bar app;
- использует bundled `ffmpeg` и `ffprobe`, без зависимости от системной установки;
- исходник не уходит на удаленный сервер;
- helper хранит source/job state локально и после рестарта может дозагрузить уже готовый хвост;
- если загрузка одного финального файла падает, повторять нужно только недогруженный хвост;
- export-session без активности более 24 часов автоматически получает статус `ABANDONED`.

Legacy fallback:
- старый поток bundle `001.mp4`, `002.mp4`, `003.mp4` и отдельный `video-processor` сохранены только для регрессий и диагностики;
- партия не может перейти в `IN_STOCK`, пока есть активный локальный export session или legacy video job.

Переход в `IN_STOCK` разрешен только если у каждого `Item` в партии есть:
- `item_photo_url`
- `item_video_url`

При переводе партии в `IN_STOCK` система автоматически:
- переводит новые `Item` в `STOCK_ONLINE`;
- считает их доступными для продажи;
- делает публичный шаблон фактически доступным на витрине, если он опубликован.

## 7. Цифровой двойник

Цифровой двойник строится из данных `Item` и связанного `Product`.

Публичные точки входа:
- страница: `/clone/:publicToken`
- QR PNG: `/api/public/items/:publicToken/qr`
- JSON API: `/api/public/items/:publicToken`

Паспорт показывает:
- серийный номер;
- дату и время сбора;
- фото и видео конкретного экземпляра;
- статус продажи;
- название и описание товара-шаблона;
- название локации и описание локации.

QR на упаковке всегда ведет на паспорт конкретного `Item`.

## 8. Продажа и остатки

На сайте продается не шаблон как абстракция, а конкретный доступный экземпляр из пула `Item`.

В v1:
- витрина агрегирует карточки по `Product`;
- остаток считается по `Item`;
- checkout разрешен только если шаблон опубликован и есть доступный `Item`.

Флаг `is_sold`:
- по умолчанию `false`;
- меняется на `true` при продаже/активации;
- влияет на публичный остаток.

## 9. Роли

- `ADMIN` — полный контроль над шаблонами, заказами на сбор, партиями, публикацией, складом и пользователями.
- `MANAGER` — HQ-операции: товары, локации, приемка, партии, склад.
- `SALES_MANAGER` — очередь заказов с сайта и статусы интернет-заказов.
- `FRANCHISEE` — принятие и выполнение заказов на сбор, просмотр своих партий, QR и финансов.
- `USER` — публичный каталог, buyer-аккаунт, checkout и история своих заказов.

## 10. Legacy-совместимость

Для обратной совместимости сохранены legacy-операции:
- ручное создание партии;
- ручное добавление item в партию;
- старый HQ acceptance flow и QR-flow.

Они нужны для регрессий и старых e2e-проверок, но основной бизнес-поток v1 должен проходить через:

`Product -> CollectionRequest -> Batch -> Item`
