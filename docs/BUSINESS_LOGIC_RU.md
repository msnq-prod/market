# Бизнес-логика системы отслеживания уникальных товаров (Stones)

Этот документ описывает полную бизнес-логику, реализованную в системе на текущий момент. Система разделена на три основных модуля: **Partner Core (Партнеры)**, **Admin HQ (Администрация)** и **Public Front (Публичная часть)**.

## 0. Жизненный цикл заказа с сайта

1. Конечный покупатель добавляет товар в корзину на публичной витрине.
2. Для оформления заказа он:
   - либо входит в существующий buyer-аккаунт;
   - либо регистрирует buyer-аккаунт по `username + password` без подтверждения почты;
   - Telegram-авторизация пока отображается только как заглушка.
3. После авторизации покупатель заполняет адрес доставки, телефон, email и комментарий.
4. Нажатие кнопки оплаты пока не проводит реальный платеж, а создает заявку-заказ со статусом `NEW`.
5. Заявка попадает в очередь `/admin/orders`, где `ADMIN` или `SALES_MANAGER` переводят ее в:
   - `IN_PROGRESS`;
   - `COMPLETED`;
   - `CANCELLED`.

## 1. Жизненный цикл Партии (Batch)

Партия — это контейнер для группы уникальных товаров, создаваемый Франчайзи (партнером) в месте добычи/производства.

### Создание Партии (Partner Core)
1.  **Авторизация**: Франчайзи входит в систему (`/partner/login`).
2.  **Начало**: Нажимает "New Batch".
3.  **Геолокация**: Указывает GPS координаты места сбора (широта/долгота).
4.  **Видео-подтверждение**: Загружает общее видео локации/процесса добычи.
    *   Видео сохраняется на сервере в `/uploads/videos`.
    *   Партия создается со статусом `DRAFT` (Черновик).

### Оцифровка Товаров (Partner Core)
1.  **Добавление фото**: В рамках черновика партии, Франчайзи загружает фотографии каждого камня/товара.
2.  **Присвоение временного ID**: Каждому фото присваивается временный идентификатор (например, номер пакетика/стикера), который наклеивается на физический товар.
3.  **Генерация Public Token**: Система автоматически генерирует уникальный, неизменяемый публичный токен (хэш) для каждого товара. Этот токен будет использоваться покупателем для проверки подлинности.
4.  **Статус товара**: Товар создается со статусом `NEW` (Новый).

### Отправка Партии (Partner Core)
1.  После добавления всех товаров, Франчайзи нажимает "Send to HQ".
2.  Статус партии меняется на `TRANSIT` (В пути).
3.  Редактирование партии блокируется.

## 2. Логистика и Приемка (Admin HQ)

Этот этап происходит на складе (HQ), когда физическая посылка прибывает от Франчайзи.

### Проверка (Acceptance Interface)
1.  **Поиск Партии**: Кладовщик вводит ID партии (из накладной) в интерфейсе `/admin/acceptance`.
2.  **Сканирование**: Берет товар в руки, сканирует временный ID (штрихкод на пакетике).
3.  **Сверка**: Система показывает фото товара, загруженное Франчайзи.
4.  **Решение**:
    *   **Принять (Accept)**: Если товар соответствует фото и качеству. Статус товара меняется на `STOCK_HQ` (На складе HQ).
    *   **Отклонить (Reject)**: Если брак или подмена. Статус товара меняется на `REJECTED`. Создается запись в аудите.

### Завершение Приемки
1.  Когда все товары обработаны, Администратор нажимает "Finish Batch".
2.  Статус партии переходит в `FINISHED`.

## 3. Распределение (Allocation)

Товары со статусом `STOCK_HQ` готовы к продаже. Администратор решает, куда они отправятся.

### Каналы Продаж (Allocation Interface)
Администратор выбирает товары в интерфейсе `/admin/allocation` и назначает канал:

1.  **Online Marketplace (`MARKETPLACE`)**:
    *   Товар остается на складе HQ физически (или на фулфилменте).
    *   Статус меняется на `STOCK_ONLINE`.
    *   Товар становится доступен для покупки на публичном сайте.

2.  **Offline Point / Consignment (`OFFLINE_POINT`)**:
    *   Товар отправляется в конкретный розничный магазин (франчайзи).
    *   Требуется указать ID получателя (Target User).
    *   Статус меняется на `ON_CONSIGNMENT` (На реализаци).
    *   Товар физически перемещается в магазин.

## 4. Финансы и Продажа (Financials)

Система автоматически учитывает финансовые операции при активации товара (продаже).

### Активация Товара (Public / Sale)
Активация происходит, когда конечный покупатель получает товар. Это может быть сканирование QR-кода внутри упаковки или API-вызов при онлайн-покупке.

1.  **Сценарий А: Розничная точка (Offline)**
    *   Товар имеет статус `ON_CONSIGNMENT`.
    *   Покупатель покупает камень в магазине.
    *   Магазин/Покупатель активирует товар (`POST /api/public/items/:token/activate`).
    *   **Финансы**: Система списывает Роялти (комиссию) с баланса Франчайзи-магазина.
        *   Операция: `ROYALTY_CHARGE`.
        *   Баланс Франчайзи уменьшается.

2.  **Сценарий Б: Онлайн продажа (Online)**
    *   Товар имеет статус `STOCK_ONLINE`.
    *   Покупатель оплачивает заказ на сайте.
    *   **Финансы**: Система начисляет выручку поставщику камня (владельцу партии), за вычетом комиссии платформы.
        *   Операция: `SALES_PAYOUT`.
        *   Баланс владельца партии (франчайзи-добытчика) увеличивается.

## 5. Роли и Доступ

*   **ADMIN (Администратор)**: Полный доступ ко всем интерфейсам (`/admin`), управление пользователями, складом, финансами и продажами.
*   **MANAGER (Операционный менеджер)**: Работа с HQ-операциями, приемкой, распределением и складом.
*   **SALES_MANAGER (Менеджер продаж)**: Доступ только к очереди интернет-заказов `/admin/orders`.
*   **FRANCHISEE (Франчайзи)**: Доступ только к Партнерскому кабинету (`/partner`). Может создавать партии, видеть свои финансы.
*   **PUBLIC / USER (Покупатель)**: Доступ к публичному сайту, каталогу товаров, карте происхождений и checkout. Для оформления заказа использует buyer-аккаунт.

## Техническая реализация (Стек)
*   **Backend**: Node.js + Express
*   **Database**: MySQL + Prisma ORM
*   **Frontend**: React + TailwindCSS (Vite)
*   **Media**: Локальное хранилище (`public/uploads`)
